# Albine Messenger — дорожная карта

Приватный E2E-мессенджер для узкого круга. Flutter Web (PWA) + Supabase.
Живёт на https://geniofalco.github.io/albine-messenger/ (автодеплой при пуше в `master`).

Документ отражает **текущее реальное состояние** — обновляется по ходу.

---

## ✅ Сделано (M1 + инфраструктура)

- **Auth**: вход/регистрация по email + паролю (Supabase Auth). Инвайт-кодов нет.
- **E2E-шифрование личных чатов**: X25519-ключи, `crypto_box` (libsodium). Сервер видит только шифротекст.
- **Хранение ключа**: приватный ключ лежит локально, зашифрован Argon2id от пароля аккаунта. Отдельного «кода восстановления» нет — на новом устройстве ключ создаётся заново (старая история на нём недоступна), как в обычных мессенджерах.
- **Личные чаты**: текст, доставка в реальном времени (Supabase Realtime).
- **Поиск людей**: по части имени пользователя/отображаемого имени → начать личный чат.
- **Дизайн**: чистая светлая тема, синий акцент. Десктоп — 2 колонки (список + чат, профиль внизу списка), мобилка — один экран со списком чатов.
- **CI/CD**: пуш в `master` → GitHub Actions собирает и публикует на GitHub Pages.

## ✅ M1.5 — Зашифрованный бэкап ключа + forward secrecy (код готов, ждёт применения миграций)

Закрывает два принятых в v1 ограничения (см. «Приняты сознательно» ниже): нет мультидевайса и нет forward secrecy. Matrix сознательно не взяли — оба свойства достигнуты точечно на текущем стеке (libsodium + Supabase + `libsignal_protocol_dart`), без замены протокола.

**1. Зашифрованный бэкап приватного ключа на сервере** — `supabase/migrations/0002_key_backup.sql`, `lib/data/key_backup_repository.dart`, изменения в `session_controller.dart` (`unlock`/`setUpProfile`).
- Отдельная таблица `key_backups` (не колонка в `profiles` — та читается всеми аутентифицированными для поиска людей, это утекло бы чужие ключи), RLS только для владельца.
- `unlock()`: локальный ключ → серверный бэкап → и только если бэкапа нет вообще — генерация нового (с загрузкой бэкапа на будущее). Неверный пароль при существующем бэкапе — явная ошибка, без побочной генерации нового ключа.
- Сброс пароля через email (`resetLocalKeyAndUnlock`) теперь явно помечает старый бэкап как недоверенный (`trustServerBackup: false`), иначе новый пароль ошибочно отклонялся бы попыткой распаковать бэкап старым паролем.

**2. Forward secrecy для 1:1 чатов через `libsignal_protocol_dart`** — `supabase/migrations/0003_signal_prekeys.sql`, `lib/services/signal/` (`signal_local_store.dart`, `signal_service.dart`), `lib/data/signal_directory_repository.dart`, интеграция в `chat_repository.dart`.
- Пакет [`libsignal_protocol_dart`](https://pub.dev/packages/libsignal_protocol_dart) (MixinNetwork) — чистый Dart, без FFI, работает на Flutter Web. X3DH + Double Ratchet + XEdDSA.
- Тот же X25519 identity-ключ, что и у legacy crypto_box, служит и Signal identity key (XEdDSA умеет подписывать Curve25519-ключом) — второго ключа для бэкапа нет.
- Новые исходящие 1:1-сообщения используют ratchet-сессию, если у собеседника уже опубликован бандл (signed prekey + registration id); если нет — грациозный откат на `crypto_box` для этой отправки. Старые сообщения остаются читаемы как раньше (колонка `messages.protocol`).
- Double Ratchet принципиально не расшифровывает свой же исходящий шифротекст (ключ сообщения удаляется сразу после использования — это и есть forward secrecy). Свои отправленные `signal`-сообщения поэтому кэшируются локально в открытом виде в момент отправки (`chat_repository.dart`, `_rememberSentEcho`/`_ensureEchoLoaded`) — персистентно в `shared_preferences` (переживает перезагрузку страницы), с ограничением на последние 500 записей. На **другом** устройстве, никогда не видевшем этот текст, он всё равно не откроется — это ожидаемо, как и у самого Signal.
- Доверие идентичности — trust-on-first-use (как и раньше, без проверки «кодов безопасности»).

**3. Живое тестирование нашло серьёзную регрессию — исправлено.** Выход из аккаунта и повторный вход ломал расшифровку у ОБОИХ собеседников. Причина: `signal_local_store.dart` хранит сессию с каждым собеседником в одном общем JSON-блоке без блокировки, а два независимых места кода (список чатов и открытый чат) могли одновременно его читать-переписывать и затирать изменения друг друга — Double Ratchet не прощает такой рассинхронизации (ключи сообщений одноразовые и удаляются сразу). Исправлено:
- `signal_service.dart` — все операции с сессией конкретного собеседника (и с собственным бутстрапом) теперь идут строго по очереди через статическую блокировку (`_withLock`/`_locks`), а не параллельно.
- Самолечение: при неудачной расшифровке сессия с этим собеседником сбрасывается автоматически (`resetSessionWith`) — конкретное сообщение теряется (это ожидаемо, forward secrecy), но переписка сама восстанавливается со следующего сообщения, без ручных действий.
- Групповой чат: если ключ группы не расшифровался, теперь показывается явная ошибка вместо вечной надписи «Расшифровка…».

**4. Пароль больше не спрашивается на каждой перезагрузке.** Расшифрованный ключ теперь кэшируется локально (`key_storage.dart`: `saveUnlockedSecretKey`/`loadUnlockedSecretKey`) и переживает перезагрузку страницы — пароль нужен только при первом входе на устройстве. Единственное, что его стирает — явное «Выйти из аккаунта» (`signOut()`); переписка при этом не теряется, следующий вход восстанавливает всё тем же паролем через серверный бэкап ключа. Осознанный компромисс (проговорён и принят): взлом самого браузера/устройства теперь может прочитать переписку без пароля — та же модель, что у WhatsApp/Telegram/Signal Web.

**5. «Сбросить ключ шифрования»** (профиль → раздел «Безопасность», по аналогии с «связанными устройствами» у WhatsApp) — на случай подозрения на взлом устройства:
- Генерирует новый identity-ключ, публикует его публичную часть — все новые сообщения/сессии сразу используют его.
- Полностью сбрасывает Signal-сессии со всеми (`SignalLocalStore.resetAll`) — вдруг скопировали не только ключ, но и состояние сессий.
- Старый ключ архивируется **только локально** (`key_storage.dart`: `addRetiredSecretKey`/`loadRetiredSecretKeys`, не бэкапится на сервер — осознанно, чтобы взломанный ключ окончательно терял доступ к серверному восстановлению) — старые сообщения остаются читаемыми на этом устройстве (`chat_repository.dart` перебирает архивные ключи при неудаче с текущим), но не на новом устройстве, если это было единственное, где ключ архивировался.
- **Исправлено:** ротация изначально не перепечатывала групповые ключи под новый публичный ключ — все группы переставали открываться сразу после сброса (`crypto_box_seal_open` не может открыть коробку, запечатанную под старый ключ, и падал необработанной ошибкой при отправке). `rotateIdentityKey` теперь перепечатывает ключ каждой группы под новый ключ до удаления старого (`_reSealGroupKeys`). Заодно `chat_screen.dart` теперь ловит любую ошибку отправки и показывает её вместо того, чтобы молча стирать введённый текст.
- **Найден и исправлен более фундаментальный баг**, который, скорее всего, стоял за большей частью «не удалось расшифровать» на протяжении всего тестирования M1.5: RPC-функция `claim_one_time_prekey` (`0003_signal_prekeys.sql`) падала на каждом вызове с ошибкой Postgres 42702 (`column reference "key_id" is ambiguous`) — `RETURNS TABLE(key_id int, ...)` неявно объявляет `key_id` как выходной параметр на весь блок функции, и `key_id` без явного префикса таблицы в DELETE стал неоднозначным. Это тихо ломало установку X3DH-сессии всегда, когда собеседник ещё не потерял one-time prekey. Исправлено в `0005_fix_claim_one_time_prekey.sql` (**требует применения к продовому Supabase**) + `signal_directory_repository.dart` теперь не падает целиком, если сам claim не удался (one-time prekey не обязателен для X3DH — деградирует до сессии без него).

**Требует применения миграции `0005_fix_claim_one_time_prekey.sql`** к продовому Supabase-проекту (SQL Editor) — это ещё не сделано; `0002`/`0003`/`0004` уже применены.

Не проверено вживую (браузерная автоматизация в этой среде не смогла отрисовать/кликнуть Flutter-web canvaskit-рендер) — нужна ручная проверка по сценариям в конце этого раздела перед тем как считать M1.5 полностью закрытым.

## ✅ M2 — Группы (код готов, ждёт применения миграции + ручной проверки)

- Создание группового чата: имя + мультивыбор участников (`lib/features/conversations/new_group_sheet.dart`, новая кнопка рядом с «Новый чат» в `conversations_screen.dart`). Групповой ключ (`crypto_service.dart`'s `generateGroupKey`) генерируется на клиенте и запечатывается (`crypto_box_seal`/`sealGroupKeyForMember`) под публичный ключ каждого участника **и себя** — сервер видит только уже запечатанные ключи.
- Атомарное создание — `supabase/migrations/0004_group_conversations.sql`, RPC `create_group_conversation` (`security definer`, по той же причине, что и `create_direct_conversation`: RLS не даёт вставить чужие строки участников до появления строки владельца).
- Отправка/приём — `chat_repository.dart`: `sendGroupMessage`/`decryptText`-ветка для `kind == group`, используют `encryptGroupMessage`/`decryptGroupMessage` (XChaCha20-Poly1305, тот же общий ключ на всех). Ключ каждого участника расшифровывается один раз и кэшируется в памяти (`_groupKeyCache`/`_tryGroupKeyFor`/`_prewarmGroupKey`) — та же схема "асинхронный прогрев перед синхронным чтением UI", что уже использовалась для Signal-кэша.
- Заодно исправлен скрытый баг: `fetchConversations()`/`fetchConversationSummary()` раньше схлопывали всех "остальных участников" в одну запись (последний в списке побеждал) — не проявлялось, пока групп не было. Теперь `ConversationSummary.members` — полноценный список для групп, `peer` остаётся как раньше только для личных чатов.
- Шапка группового чата показывает «N участников» под названием.
- **Требует применения миграции `0004` к продовому Supabase-проекту** — ещё не сделано, как и `0002`/`0003` из M1.5.
- Не проверено вживую (см. ограничение браузерной автоматизации в M1.5 выше) — нужна ручная проверка: создать группу из ≥2 участников, отправить сообщения в обе стороны, убедиться что обычные личные чаты не задеты.

**Осознанно не в этом заходе** (см. «Приняты сознательно»): добавление участников в существующую группу, ротация группового ключа при исключении, forward secrecy для групп.

**Известное ограничение (не баг, отложено):** два устройства одного аккаунта (например, телефон и ПК), одновременно в сети, не видят живые сообщения друг друга в forward-secret личных чатах — каждое строит свою независимую Double Ratchet-сессию с собеседником, и шифротекст одной сессии принципиально не открывается другой. Так и должно быть на уровне протокола — у WhatsApp/Signal это решается полноценным multi-device (каждое устройство — отдельный `deviceId`, отправитель шифрует отдельную копию под сессию каждого устройства, включая свои же остальные). Это большая отдельная фича; пока сознательно отложена — история между устройствами по-прежнему синхронизируется через серверный бэкап ключа, просто не в реальном времени одновременно с двух устройств.

## ✅ M2.5 — UI: единая кнопка чата, действия над чатом и сообщением (код готов, ждёт применения миграции + проверки)

Пауза перед M3 по просьбе пользователя — интерфейс "почти пустой", нужны стандартные вещи из любого мессенджера.

- **Одна кнопка вместо двух**: «Новый чат»/«Новая группа» в `conversations_screen.dart` заменены на одну ("+"), открывающую шторку с выбором.
- **Список чатов**: свайп вправо — закрепить, свайп влево — отключить звук/удалить, долгое нажатие — то же одним меню (`flutter_slidable`). Закреп/мьют/«удаление» — по `conversation_members.pinned_at/muted/hidden_at`, своя запись на каждого участника, RLS не менялся (уже разрешено обновлять свою строку членства). «Удаление» чата — локально для аккаунта: чат прячется из списка, новое входящее сообщение возвращает его обратно (как у WhatsApp/Telegram), у собеседника ничего не меняется.
- **Действия над сообщением** (долгое нажатие): Ответить, Закрепить, Переслать — всем участникам; Редактировать/Удалить — только своих сообщений.
  - Закрепление — через RPC `toggle_message_pin` (любой участник может закрепить/открепить, но RPC трогает только `pinned_at`, подменить чужой контент через него нельзя).
  - Пересылка — расшифровывается локально и шифруется заново для целевого чата (это независимое новое сообщение, не ссылка на старый шифротекст — целевой чат может быть на другом протоколе/ключе).
  - Удаление — мягкое (`deleted_at`), но и шифротекст затирается на сервере, а не просто помечается.
  - **Редактирование сделано как отдельное новое зашифрованное сообщение** (`edits_message_id` → id исходного), а не перезапись шифротекста на месте — у `signal`-сообщений ключ уже уничтожен после использования (та же причина, по которой свои исходящие signal-сообщения нельзя расшифровать даже себе), перезапись задним числом — тот же класс рассинхрона ratchet, что мы уже чинили. Так же это устроено и у настоящего Signal/WhatsApp.
- Новая миграция `supabase/migrations/0006_conversation_message_actions.sql` — новые колонки в `conversation_members`/`messages` + RPC `toggle_message_pin`. **Требует применения к продовому Supabase** — не сделано, как и `0005`.
- Не проверено вживую — нужна ручная проверка: свайпы/долгое нажатие в списке, ответ/редактирование/закреп/пересылка/удаление сообщения в личном и групповом чате, что скрытый чат возвращается при новом сообщении.

## 🧭 Дальше / открытые вопросы

- **Доступ из России**: Supabase Cloud (AWS/Cloudflare) подпадает под блокировки → нужен self-host Supabase на российском VPS. Код и схема к этому готовы (меняется только URL).
- **Нативные приложения** (iOS/Android из того же Flutter-кода): дадут полноценные пуши «как у VK/MAX» без ограничений PWA. Требует аккаунтов разработчика (Apple $99/год, Google $25) и ревью в сторах.
- **Осознанный выбор модели приватности**: строгий E2E (как Signal) vs «сервер читает» (как VK) — влияет на уведомления, синхронизацию истории, серверный поиск.

## Приняты сознательно (v1)

- Ротация групповых ключей при исключении участника, и вообще добавление участников в уже существующую группу — остаются отложенными (группа создаётся сразу с финальным составом).
- Forward secrecy распространяется только на личные чаты (M1.5) — группы (M2) на общем симметричном ключе без ratchet.
- Нет проверки «кодов безопасности» (safety numbers) — идентичность собеседника/группы принимается по first-use, как и раньше.
